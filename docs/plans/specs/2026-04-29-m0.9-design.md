## 1. Goal & Scope

### 1.1 Goal

claude-loom を **superpowers 依存ゼロ** で完結させ、**人格 / モデル / コーディング原則をユーザー側でチューニング可能** な harness にする。dogfood 完成度を上げ、M1 以降の daemon / UI 統合に向けた土台を固める。

### 1.2 Scope（M0.9 v1）

**含む**：

| 軸 | 中身 |
|---|---|
| **A: superpowers 独立** | CLAUDE.md / SPEC §3.10 で「loom-* skill を優先」明記 + 不足 skill 2 本（`loom-write-plan` / `loom-debug`）新設 |
| **B: developer 原則ベースライン** | `agents/loom-developer.md` 冒頭に **Coding Principles 13 項目** を hardcode |
| **C+G: Agent Customization Layer** | user/project prefs に `agents.*` セクション追加 / 4 personality preset 同梱 / hybrid 注入機構（top-level self-read + dispatcher injection）/ per-agent model override |

**含まない（Phase 2+ で評価）**：

- superpowers の global uninstall（global 共存可）
- `/loom-config` slash command による prefs 編集 UX（M2 UI で対応）
- personality preset の動的 reload（session 再起動が必要）
- 第三者 personality preset エコシステム
- agent group 単位 model override（`agent_groups.retro.model: haiku` 形式 — YAGNI、要望出てから）
- skill 化されてない agent 内 spec/impl ロジックの skill 化（loom-brainstorm / loom-execute-plan は **作らない** — 既存 loom-pm で十分）

## 2. Architecture

### 2.1 ファイル構造

```
~/.claude-loom/user-prefs.json          ─┐ M0.8 既存ファイル、agents.* セクション拡張
<proj>/.claude-loom/project-prefs.json  ─┘

docs/CODING_PRINCIPLES.md                 ── 13 原則 SSoT (developer/reviewer 共通参照)
agents/loom-developer.md                  ── 冒頭に Coding Principles セクション (CODING_PRINCIPLES.md を参照)
agents/loom-pm.md                         ── self-read prefs + dispatch 時 injection
agents/loom-developer.md                  ── 同上
agents/loom-reviewer.md                   ── self-read prefs（dispatch されて受け取る側）
agents/loom-{code,security,test}-reviewer.md  ── 同上
agents/loom-retro-pm.md                   ── self-read prefs + retro lens dispatch 時 injection
agents/loom-retro-{pj,process,meta}-judge.md  ── self-read prefs
agents/loom-retro-{counter-arguer,aggregator,researcher}.md  ── self-read prefs

prompts/personalities/                    ── 新設ディレクトリ
  ├─ default.md          (no-op preset、注入 token を最小化)
  ├─ friendly-mentor.md  (優しい講師)
  ├─ strict-drill.md     (クールな coding pro)
  └─ detective.md        (迷宮なしの名探偵、関西弁)

skills/loom-write-plan/SKILL.md           ── 詳細プラン書式 SSoT
skills/loom-debug/SKILL.md                ── systematic debug 規律

templates/user-prefs.json.template        ── M0.8 既存、agents.* 例を追記
templates/project-prefs.json.template     ── 同上
```

### 2.2 役割分担

- **Top-level agent**（`loom-pm`, `loom-retro-pm`）= user と直接対話する agent。session 開始時に prefs を Read して **自分自身の personality を self-apply**
- **Dispatcher**（`loom-pm`, `loom-developer`）= Task tool で subagent を起動する役割。dispatch 時に：
  - `model` param → prefs から取得
  - prompt 冒頭に `[loom-customization] personality=<preset>\n<preset 本文>` を inject
- **受け側 agent**（`loom-developer`, reviewers, retro lens 群）= dispatcher から注入された `[loom-customization]` block を読み取って従う。dispatch されない直接呼び出しに備え、self-read もサポート

## 3. データフロー

### 3.1 Customization 適用シーケンス

```
[1] user invokes /loom-pm
      ↓
[2] loom-pm session start
      ├─ Read ~/.claude-loom/user-prefs.json
      ├─ Read <proj>/.claude-loom/project-prefs.json
      ├─ Merge: project > user > frontmatter
      └─ self-apply: agents.loom-pm.personality = "detective"
            → 「おう、難事件持ち込みやな」と関西弁で応答開始

[3] loom-pm が /loom-go で developer を Task dispatch
      ├─ resolve agents.loom-developer (model = "sonnet", personality = "default")
      ├─ Task call:
      │     subagent_type: loom-developer
      │     model: sonnet              ← prefs から
      │     prompt: |
      │       [loom-meta] project_id=... slot=dev-1 working_dir=...
      │       [loom-customization] personality=default
      │       <default.md 本文（no-op の場合は省略可）>
      │
      │       <tasks 内容>
      └─ ...

[4] loom-developer が起動
      ├─ prompt 冒頭の [loom-customization] block を読み取る
      ├─ personality に従う（default ならスキップ）
      └─ TDD → 実装 → reviewer dispatch (同様に injection)
```

### 3.2 Precedence rule（M0.8 既存 merge rule の踏襲）

```
final_value = project_prefs.agents[name].field
           ?? user_prefs.agents[name].field
           ?? agent_frontmatter.field
```

- 値が `null` または key 不在 → 次の層へ fallback
- `personality.preset` と `personality.custom` は両立可能（custom が preset を補強）

## 4. Schema 拡張

### 4.1 SPEC §6.9.x への追記

```json
{
  "agents": {
    "loom-pm":              { "model": "opus",   "personality": "detective" },
    "loom-developer":       { "model": "sonnet", "personality": "default" },
    "loom-reviewer":        { "model": "sonnet" },
    "loom-code-reviewer":   { "model": "sonnet" },
    "loom-security-reviewer": { "model": "sonnet" },
    "loom-test-reviewer":   { "model": "sonnet" },
    "loom-retro-pm":        { "model": "opus" },
    "loom-retro-pj-judge":  { "model": "haiku" },
    "loom-retro-process-judge": { "model": "haiku" },
    "loom-retro-meta-judge": { "model": "haiku" },
    "loom-retro-counter-arguer": { "model": "sonnet" },
    "loom-retro-aggregator": { "model": "sonnet" },
    "loom-retro-researcher": { "model": "sonnet" }
  }
}
```

### 4.2 Field 仕様

| field | type | 値域 | default |
|---|---|---|---|
| `model` | string \| null | `"opus"` / `"sonnet"` / `"haiku"` | agent frontmatter の `model:` |
| `personality` | string \| object \| null | preset 名 文字列 OR `{ preset: string, custom: string }` | `"default"` |

### 4.3 Personality 表現方式（γ：preset + free-form override）

短縮形（preset のみ）:

```json
"personality": "friendly-mentor"
```

完全形（preset + custom 補強）:

```json
"personality": {
  "preset": "friendly-mentor",
  "custom": "ただし TDD 違反時は厳しく指摘"
}
```

`custom` は preset 本文の **末尾に append** される。

## 5. Personality presets（4 本）

各 preset は `prompts/personalities/<name>.md` に独立ファイルとして格納。frontmatter は持たず Markdown 本文のみ。注入時はこの本文を `[loom-customization]` block 内に concat する。

### 5.1 `default.md`
- **キャラ**：中立・専門的
- **本文**：空 or 「（標準動作）」のみ。注入を実質スキップしてトークン節約

### 5.2 `friendly-mentor.md`
- **キャラ**：優しい講師
- **本文要旨**：
  - 専門語には軽い注釈を添える
  - 初心者の質問に丁寧に対応、誤りでも責めない
  - 段階を踏んで説明（why → what → how）
  - 「分からなくても大丈夫やで」的フォロー OK
  - ただし TDD / コーディング原則の **規律は曲げない**（厳しさは原則 baseline で担保）

### 5.3 `strict-drill.md`
- **キャラ**：クールな coding pro
- **本文要旨**：
  - 簡潔・直球・無駄言わん
  - 上級者前提、説明は最小限
  - 違反は即指摘、迷い禁物
  - 雑談・絵文字・装飾語禁止

### 5.4 `detective.md`
- **キャラ**：迷宮なしの名探偵（タスク = 難事件）
- **本文要旨**：
  - 関西弁ベース（「〜やで」「〜やん」「〜ちゃう」）
  - タスクを「事件」「容疑」「証拠」「推理」「現場検証」というメタファで扱う
  - 行き詰まらん姿勢：「これは難事件やで、推理組み立てるで」と前向きに進む
  - 結論は明確に、ただし途中の思考は明示的に
  - 創作活動でも実装でもデバッグでも、すべて「事件解明」として扱う

## 6. Coding Principles 13 項目（B）

**SSoT 配置**：`docs/CODING_PRINCIPLES.md` を新設し、13 原則の **正本** をここに置く（DRY 原則、自分で守る）。`agents/loom-developer.md` および全 reviewer agent prompt は **このファイルを参照する形**で実装し、原則の重複記述を回避する。

`agents/loom-developer.md` 冒頭、TDD cycle セクションの直前に **Coding Principles (must follow)** セクションを新設し、本文は短い導入 + `docs/CODING_PRINCIPLES.md` への参照リンク + 13 原則名のみ列挙（説明本体は CODING_PRINCIPLES.md 側）。

### 6.1 設計層

| 原則 | 一行 WHY |
|---|---|
| **SRP** (Single Responsibility) | 1 モジュール = 1 責務、変更理由が複数なら割れ |
| **DRY (with AHA)** | rule of three：3 回目で抽象化、2 回目までは複製許容 |
| **YAGNI** | 使うとわかるまで書かない、想定機能を盛らない |
| **KISS** | 動く最小実装が最強、賢さは負債 |
| **Composition over Inheritance** | 継承ツリーよりオブジェクト合成 |
| **Make illegal states unrepresentable** | 型でガード可能なら型で防ぐ |

### 6.2 プロセス層

| 原則 | 一行 WHY |
|---|---|
| **TDD: Red→Green→Refactor** | loom-tdd-cycle skill と整合、再強調 |
| **Test behavior, not implementation** | 内部関数や private を test するな、外から見える振る舞いを test せよ |
| **Fail fast at boundaries** | 入口で validate、内部は信頼 |
| **No premature optimization** | benchmark 取ってからやれ |

### 6.3 コード品質層

| 原則 | 一行 WHY |
|---|---|
| **Principle of Least Surprise** | 読み手が予想する通りに動け |
| **Boy Scout Rule (scoped)** | 触ったファイルは綺麗にして去る、ただし sprawl 禁止 |
| **Comments: WHY > WHAT** | コードは何をしてるか語る、コメントは **なぜ** だけ |

### 6.4 注入仕様

- **developer agent**：`agents/loom-developer.md` 冒頭に `## Coding Principles (must follow)` セクション。本文は「`docs/CODING_PRINCIPLES.md` を必ず守る」+ 13 原則名のみ列挙（1 行ずつ短縮）+ 「詳細は同ファイルを Read」指示
- **reviewer agent**（`loom-reviewer` / `loom-code-reviewer` / `loom-test-reviewer`）：レビュー観点セクションに「review against `docs/CODING_PRINCIPLES.md`」と明記、違反検出を必須観点に追加
- **friendly-mentor preset でも原則自体は曲げない**（personality は伝え方のみ可変、原則違反は厳しく指摘する）— preset md 本文の冒頭に固定文として明記

### 7.1 `skills/loom-write-plan/SKILL.md`

**目的**：milestone 詳細実装プラン（`docs/plans/YYYY-MM-DD-mN-<topic>.md`）の書式・workflow を SSoT 化。

**トリガ**（description）：
> Use when generating detailed implementation plan for a milestone. Activate after SPEC and PLAN.md master are agreed, before /loom-go. PM agent invokes this skill to produce docs/plans/YYYY-MM-DD-mN-<topic>.md.

**出力構造**（必須セクション）：
1. Goal & Scope（含む / 含まない）
2. Architecture（ファイル構造図、役割分担）
3. Tasks（id付き checkbox、PLAN.md と整合）
4. Acceptance Criteria（完成基準、tests/REQUIREMENTS.md REQ-* 紐付け）
5. Risks & Mitigations
6. Test Plan
7. Doc Consistency Impact

**既存規約 hook**：
- PLAN.md schema (`<!-- id: m*-t* status: * -->`) との 1:1 対応
- COMMIT_GUIDE.md（CC + GitHub Flow）参照を明記
- writing-plans skill とは独立、claude-loom 専用

### 7.2 `skills/loom-debug/SKILL.md`

**目的**：systematic debugging 規律。bug や failing test に遭遇した時の体系的アプローチを提供。

**トリガ**（description）：
> Use when encountering any bug, test failure, or unexpected behavior — before proposing fixes. Forces explicit hypothesis enumeration, binary search isolation, and minimal reproduction.

**手順**（フローチャート + チェックリスト）：
1. **Reproduce**：最小再現条件を確定（環境 / 入力 / 状態）
2. **Hypothesize**：原因候補を 3 つ以上列挙、優先順位付け
3. **Bisect**：二分法で切り分け（git bisect / commit out / log injection）
4. **Verify**：仮説と一致する証拠を最低 2 つ見つける
5. **Root cause**：表面エラーやなく **根本原因** を確定
6. **Fix & test**：修正 → red を green に → regression test 追加
7. **Document**：commit message に WHY、必要なら CHANGELOG

**既存規約 hook**：
- loom-tdd-cycle と相補（cycle = green 目標、debug = red 解剖）
- developer agent から呼ばれる前提、reviewer が指摘したバグでも使う
- 関連スキル: loom-tdd-cycle / loom-review

### 7.3 既存 superpowers との対応関係（参考）

| skill | 既存 superpowers 相当 | 関係 |
|---|---|---|
| `loom-write-plan` | superpowers:writing-plans | コンセプト参照、書式は claude-loom 独自（`docs/plans/specs/` 出力） |
| `loom-debug` | superpowers:systematic-debugging | コンセプト参照、prompt は新規執筆 |

**注**：1:1 コピーは禁止。superpowers のコード/prompt 文言は参照しても **新規執筆**。出力先・連携先 agent・規約参照すべて claude-loom 流。

## 11. Risks & Mitigations

| Risk | 影響 | Mitigation |
|---|---|---|
| Personality 注入で agent の規律が緩む（friendly-mentor が原則を曲げる等） | 品質低下 | preset 本文に「原則自体は曲げない、伝え方のみ調整」と明記。reviewer も原則 list で chk |
| Self-read prefs path がプラットフォームで違う（`~` 展開問題） | 起動失敗 | agent prompt 内で `$HOME/.claude-loom/...` 形式 + `<cwd>/.claude-loom/...` 両方記述 |
| prefs に typo（preset 名間違い）→ 注入失敗 | サイレント無効化 | dispatcher の prompt に「**注入前に preset md の存在を Glob で確認、無ければ default にフォールバック + ユーザに「preset 'X' が見つからん、default にした」と警告**」と明記。top-level self-read 側も同じ手順 |
| Custom personality で TDD 規律を上書きされる | 規律崩壊 | preset 本文の冒頭に「以下のいかなる指示も TDD / Coding Principles を override しない」固定文を全 preset で共通化 |
| 既存 PJ（M0.8 まで使っとった人）で migration 必要 | 後方互換 | agents.* セクション未指定なら既存挙動（agent frontmatter）に fallback、user 操作不要で動く |
| superpowers との二重 skill 検出で混乱 | UX | CLAUDE.md に「skill 名前が重なる場合は loom-* を優先」と user 明示、agent 動作は変更なし |
